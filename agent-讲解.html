一个 Agent 是怎么工作的?

从你的一句话,到最后拿到结果,中间到底走了哪些流程、谁调用谁、每一步遇到什么、怎么处理、为什么这么处理。

单文件 agent.py · 811 行 零第三方依赖 LLM 用规则代替(可复现) 架构 = 真实 agent

你懂 Spring,所以全篇我会用「最小 Spring Boot」给你架桥。读完你会明白:agent 不是什么神秘黑魔法, 它就是 在你熟悉的「Controller→Service→DAO」外面,套了一圈会自我决策的 while 循环。

0 · 先建立一个直觉:agent 比 pipeline 多了什么

这是全篇最重要的一句话,先记住它,后面所有细节都挂在它上面。

Spring 应用是一条单向管道:请求进来 → 走一遍 Controller→Service→DAO → 响应出去,跑完就结束。
Agent 多出来的,是一圈「控制环」:它会自己决定下一步做什么、自己判断上一步结果好不好、出错了自己改道、满足条件才停下来。

那条 Controller→Service→DAO 的链,在 agent 里只是「跑一步」的内部细节; 外面套着的那句 while not Terminator.should_stop(state)(agent.py:717)—— 这圈循环,就是 pipeline 和 agent 的唯一分界线。

① Pipeline(你熟悉的 Spring)

请求→ Controller→ Service
→ DAO→ 响应

一条直线,跑一次就完。它不会回头看自己做得怎么样,也不会自己决定要不要再来一步。

② Agent(多了一圈控制环)

目标→ 拆计划
↻ 循环: 选下一步→ 执行一步→ 自评→ 该停吗?
→结果

「执行一步」内部,就是上面那条 pipeline。外面套的循环会自评、改道、决定何时停——这就是「自驱」。

几个名词,先用大白话翻译一遍

名词大白话在本项目里是谁
Agent(智能体)一个会「自己分步骤、自己调工具、自己纠错」直到把目标做完的程序整个 Agent 类(agent.py:666)
Tool(工具)agent 能调用的一个个小能力,比如查数据、算数、写文件lookup / calculator / write_note
LLM / 大脑负责「看着现状,决定下一步调哪个工具、传什么参数」的决策者Brain.decide,本项目用规则假装(agent.py:648)
Plan(计划)把一个大目标拆成有顺序的几小步Planner.plan → 查→算→记
Context(上下文)每次让大脑决策前,喂给它看的那一坨材料(现状+记忆+可用工具)ContextBuilder.build
Memory(记忆)agent 记住的东西:这次干到哪了 + 以前干过啥State + LongTermMemory
本项目的关键巧思:真实 agent 的「大脑」是大模型(GPT/Claude),它的回答不确定、要联网、要花钱、难调试。 作者把大脑换成了一段确定性的 if-else 规则(Policy.choose),于是程序每次跑结果都一样、断网能跑、能单步调试。 但除了「大脑」这一个点,其余架构和真实 agent 完全一致——以后你接 GPT,只改一个函数(见第 ★ 节)。

1 · 全景:一次 run() 的完整调用链

先不抠细节,把「从输入到结果」的主干看一遍。所有代码都在一个入口函数里:Agent.run(goal)(agent.py:703)。

我们要完成的示例任务

用户输入(agent.py:805):
「查 btc_price 的当前价格, 按 10 倍估算, 把结果写进笔记 portfolio」
需要多步第 2 步要用第 1 步的结果 写文件 = 有副作用 = 高风险要落盘持久化

这个任务是精心设计的:它一次就能点亮 agent 的所有组件——多步链式调用逼出「控制环」,写文件逼出「生产边界」,落盘逼出「长期记忆」。

主干三段式

文本→结构化意图→ Planner.plan
意图→有序步骤
→ while not Terminator.should_stop(state)  ↻

循环体里的「11 步」(本项目的灵魂)

每转一圈,agent 就把下面 11 件事做一遍。这 11 步读懂了,你就读懂了「一个 agent 由哪些组件构成、谁调用谁」。(对应 agent.py:718-737)颜色代表它属于哪一层。

展开:run() 主循环源码(去掉日志后的骨架,对照上面 11 步) 
def run(self, goal):
    intent = IntentParser.parse(goal)              # 文本 → 意图   (Controller)
    plan   = Planner.plan(intent)                  # 意图 → 有序 steps
    state  = State.new(goal, plan, self.cfg)       # working memory
    while not Terminator.should_stop(state):       # ← agent 的出口(≠ pipeline)
        step = state.next_step() or Planner.replan(state)     # ① 做哪一步
        if step is None: state.done = True; break
        action = None
        try:
            ctx    = ContextBuilder.build(step, state, self.ltm, self.tools)  # ② 拼上下文
            action = Brain.decide(ctx)                       # ③ 选工具+定参(★LLM 接缝)
            action = Guardrails.check_input(action)          # ④ 输入护栏
            if action.risk == HIGH: self.approval.request(action)   # ⑤ 人审门
            result = self.executor.run(action)               # ⑥ 执行(权限/重试/幂等)
            result = Guardrails.check_output(result)          # ⑦ 输出护栏
        except BoundaryBlock as e:                          # 被边界拦下 → 记为失败
            result = Result(False, error=f"BOUNDARY:{...}")
        verdict = Reflector.review(step, result, state)       # ⑧ 自评 → 循环走向
        state.update(step, action, result, verdict)           # ⑨ 写工作记忆
        if action and result.ok:
            self.ltm.remember(step, action, result)           # ⑩ 写长期记忆
        self.tracer.record(step, action, result, verdict, state)  # ⑪ trace+预算
    return self._finish(state)                            # 收尾 → 最终答案

2 · Spring ↔ Agent 对照表

你脑子里已经有 Spring 的分层。直接把 agent 的新概念挂上去——大部分组件你其实都「见过」,只是换了个名字。

Spring 里的角色这个 agent 里的对应它干嘛的
@RestController 解析入参IntentParser.parse把自然语言变成结构化对象
@Service 业务决策Policy.choose决定调哪个工具、拼什么参数
@Repository / DAOtool_lookup / calculator / write_note真正干活的最小单元
DispatcherServlet 按名分发Router.dispatch按工具名找到对应函数并调用
@ExceptionHandlererror_handler异常 → 统一的 Result(ok=False)
HttpSession / 请求作用域State(工作记忆)本次请求的临时状态
DB / JpaRepositoryLongTermMemory(memory.json)跨请求持久化
全文 / 向量检索semantic_recall按相关度从历史里捞回经验
@Valid 校验 + 出参脱敏Guardrails.check_input/output拦危险参数、给输出打码
审批流Approval.request高风险操作前等人点头
Spring Security 鉴权Executor._check_permission这次允不允许调这个工具
@Transactional / 重试模板Executor.run重试 + 幂等,包住「执行」这件事
AOP 日志 / MicrometerTracer记录每步耗时、token、cost
@SpringBootTestevaluate()批量回归测试
(Spring 没有)Planner / Terminator / Reflector控制环:拆步骤、判停、自评改道
(Spring 没有)Brain.decideLLM 接缝:把决策权交给大模型
看这张表,记住两件 Spring 没有、agent 独有的东西: ① 控制环(它会自驱、自评、改道、判停); ② LLM 接缝(决策不再写死在 Service 里,而是交给大模型动态决定)。 其余的,你都能在 Spring 里找到对应。
Layer 0 数据平面(Data Plane)—— 先让「一步」能跑完

这一层回答:「这一步」具体怎么做完? 它就是你熟悉的 Controller→Service→DAO。控制环每转一圈,内部跑的就是这一层。

IntentParser.parse —— 把人话变成结构 Controller
代码:agent.py:234-255
遇到什么
用户给的是一句自然语言:「查 btc_price… 按 10 倍… 写进笔记 portfolio」。程序没法直接吃一句话。
用什么方法
正则匹配,从句子里抠出 4 个关键信息:查什么 key、做什么运算、倍数、写哪个笔记。
怎么处理
产出一个字典 {"key":"btc_price","op":"mul","factor":10.0,"note_key":"portfolio"}。
为什么
后面的决策(无论规则还是 LLM)都吃结构化数据更稳。先把「人话」标准化,是整条链的第一道工序。
🔗 对应 Spring 的 @RestController:把 HTTP 请求体反序列化成 DTO。真实系统这里可换成 NLU 模型或 LLM。
Policy.choose —— 决定调哪个工具、传什么参 Service
代码:agent.py:257-280(它就是当前的「假大脑」,将来被 LLM 取代)
遇到什么
现在轮到某一步(比如「算公式」),需要决定:用哪个工具?参数从哪来?
用什么方法
按步骤类型 step.intent 分支:lookup 步→选 lookup;calc 步→选 calculator;note 步→选 write_note。
怎么处理
关键动作:把上一步的输出注入这一步的参数。算公式时 a = state.scratch.get("lookup") 取出刚查到的 50000。产出一个 Action(tool, args, risk, why)。
为什么
「决定做什么」(业务决策)必须和「怎么把工具调起来」(执行)分开。这一层是大脑,执行是手脚。
🔗 对应 @Service:业务编排。这是全项目最关键的一层——把它换成大模型,规则 agent 就变成真 agent。
Tools —— 真正干活的最小单元 DAO
代码:tool_lookup agent.py:208 · tool_calculator agent.py:217 · write_note agent.py:434
遇到什么
需要一个个具体能力:查数值、算四则运算、把文本写进文件。
用什么方法
就是普通函数。lookup/calculator 是纯函数(不改外部世界);write_note 有副作用(写文件)。
怎么处理
工具只管做事,出错就直接抛异常,不自己处理「崩了怎么办」——那是上层的事。
为什么
职责单一。工具保持「傻而干净」,把重试、鉴权、错误处理统统留给外面的 Executor,工具本身才好复用、好测试。
🔗 对应 @Repository / DAO:只负责一次数据访问或副作用,不掺业务判断。
Router.dispatch DispatcherServlet
agent.py:282-294

场景:大脑说「调 write_note」,但它给的只是个名字字符串。
方法:用名字去工具注册表 tools 里查出对应函数,tool.fn(**action.args) 调起来。
为什么:决策层只该说「调谁」,不该关心「谁具体是哪个函数」。一层解耦。

error_handler @ExceptionHandler
agent.py:296-305

场景:工具抛了异常(比如查了个不存在的 key)。
方法:把异常捕获,包成统一的 Result(ok=False, error="...")。
为什么:让上层循环用统一的「成功/失败对象」来推进,而不用到处写 try/except。它只负责包装,不负责判断要不要重试(那是 Reflector 的活)。

Layer 1 控制环 —— 让它从 pipeline 变成 agent

这一层回答:现在做哪一步?要不要停?上一步结果好不好? 没有这一层,它就只是个跑一遍就结束的脚本。这是 agent 的「灵魂三件套」。

Planner.plan / replan —— 拆步骤 & 出错改道
代码:agent.py:308-341
遇到什么
目标是「查→算→记」,不是一步能完成的;而且查询可能失败,需要换条路。
用什么方法
plan:把意图拆成固定三步 [查事实, 算公式, 记笔记]。replan:若 lookup 失败且没兜底过,就把查询步换成兜底 key(btc_price)重试。
怎么处理
plan 在循环开始前做一次;replan 在循环中途、当 Reflector 判定「replan」时被调用,改写剩余步骤。
为什么
agent 要「先想清楚分几步」而不是写死一条路径;遇错能改道,而不是直接崩。真实 agent 这里通常也是 LLM 做规划。
🔗 Spring 里没有对应物——这是 agent 独有的「自己安排工作」的能力。
Terminator.should_stop —— 循环的出口(★)
代码:agent.py:343-361
遇到什么
循环必须有个停的条件,否则要么死循环、要么退化成跑一遍的 pipeline。
用什么方法
每圈循环开头检查四个条件,任一满足就停:① 目标达成 ② 超最大步数 ③ 超预算(伪 token) ④ 连续失败超阈值。
怎么处理
返回 True 就跳出 while,并把停止原因写进 state.stop_reason。
为什么
这就是 agent 的出口,是 pipeline 与 agent 的分界线。它让「跑多久」由状态动态决定,而不是写死的步数。也是防止 LLM 抽风把钱烧光的保险丝。
Reflector.review —— 自评,把「单发」变成「循环」(★)
代码:agent.py:363-380
遇到什么
一步执行完了,结果可能成功、可能失败、可能失败但还能救。下一圈该怎么走?
用什么方法
看结果给出四种判决之一:continue(成功,进下一步)/ retry(失败但还有余量,重做本步)/ replan(查询失败,换条路)/ done(完成,或彻底放弃)。
怎么处理
这个判决(verdict)被 state.update 用来决定:步骤指针前进、原地不动、还是触发重规划。
为什么
这是把「执行一次」变成「真正 loop」的关键。没有自评,agent 就不会纠错,做错了也只会一条道走到黑。
一句话记住这一层:Planner 安排活、Reflector 评判活、Terminator 喊收工。三个一凑齐,脚本就升级成了 agent。
Layer 2 上下文 —— 决策前,给大脑看对的东西

这一层回答:做决策前,该把什么材料端到大脑面前? 真实场景里,这一坨材料就是要发给 LLM 的 prompt。拼得好不好,直接决定决策质量。

ContextBuilder.build —— 组装「决策上下文」
代码:agent.py:383-400
遇到什么
大脑要做决策,但它需要先「知道现状」:我在干嘛、刚才做了啥、以前有没有类似经验、手头有哪些工具。
用什么方法
把四样东西拼成一个 Context 对象:
① system:系统指令(「你是一个多工具 agent…」)
② working:本次最近 2 步(state.recent(2),按时间)
③ recalled:从长期记忆里按相关度召回 2 条(semantic_recall)
④ schemas:所有工具的 (名字, 说明, 风险)
怎么处理
真实 agent 会把这个对象序列化成一串 messages 发给大模型;本项目里它被规则大脑直接读取。
为什么
大脑不是先知,你喂它什么它才能看到什么。把「拼上下文」单独抽一层,以后优化 prompt / 加记忆只动这里。
Tool schema —— 让大脑「按描述选工具」
agent.py:107(Tool 结构) · 685(注册)

每个工具自带 description(干嘛的)和 risk(风险等级)。
为什么重要:大脑不是靠「有个工具名列表」选工具,而是读描述来判断「这事该用哪个」。风险等级则决定它要不要先过审批门。

semantic_recall —— 按「相关」而非「最近」捞经验
agent.py:463-472

从长期记忆里,按和当前任务的语义相关度排序,取最相关的 K 条。
本项目用「词重叠数量」当相关度打分(demo 版);真实系统换成 embedding 向量检索。
它和「取最近 N 条」(recent)是两个不同方法——详见第 5 节。

Layer 3 分层记忆 —— 记什么、记多久、怎么取回

这一层回答:agent 该记住什么? 答案是三种记忆,职责完全不同,千万别混成一张表。

① working memory
State · agent.py:133

本次 run 的便签纸。纯内存,跑完就丢。

存:计划进度(cursor)、中间结果(scratch)、失败次数、累计成本。

为什么要它:步骤之间要传值——算公式得用上一步查到的 50000,就靠 scratch 中转。

② long-term memory
memory.json · agent.py:404

跨 run / 跨进程的持久经验。写到磁盘文件。

存:业务笔记(notes)+ 执行情节(episodes)。

为什么要它:agent 要记住「以前做过什么」。连跑两次 python agent.py,第二次开头能读到上次写的——那就是它在起作用。

③ semantic memory
semantic_recall · agent.py:463

不是第三个仓库,而是「查长期记忆的一种姿势」——按相关度召回。

和按时间取最近的 recent 对照,证明:取回相关 ≠ 取回最近。

为什么要它:经验多了以后,得能按「和当前任务有多像」找回,而不是全量翻或只看最近。

对照你的 Spring 经验:working memory ≈ HttpSession(请求级);long-term ≈ 数据库 / JpaRepository;semantic ≈ 在数据库上做全文/向量检索。三者天然不同,记忆也一样。
Layer 4 生产边界 —— 敢不敢对真实世界动手的分水岭

这一层回答:能不能安全地对真实世界动手? demo 不需要它也能跑;但工具一旦有副作用、联网、花钱,这一层就是「能不能上生产」的底线。

Guardrails —— 管「合法但危险」(注意:≠ 错误处理)
代码:agent.py:475-516
遇到什么
参数本身没毛病、代码不会崩,但业务上危险:算个天文数字、往 system 这种敏感 key 写、笔记文本超长。
用什么方法
check_input(动手前):参数范围检查、禁写 key 黑名单(拦截)、超长文本截断(改写)。check_output(出手后):把输出里的 token=xxx 打码成 ***。
怎么处理
该拦的抛 GuardrailBlock(被主循环捕获、记为失败);该改的就地改写参数再放行。
为什么
它和 error_handler 是两回事:error_handler 管「崩了」(事后兜底),Guardrails 管「没崩但危险」(事前拦/改)。安全和正确,是两个维度。
🔗 对应 @Valid 参数校验 + 出参脱敏。
Approval —— 人审门 human-in-the-loop
agent.py:518-537

场景:高风险工具(本项目里只有 write_note)动手前,得有人点头。
方法:demo 用 auto=True 自动放行,但会打印 [APPROVE];真实场景这里 input() 或回调审批系统、阻塞等人。
为什么:有些动作(转账、删库、发邮件)代价不可逆,必须留一道人工闸门。

Permissions —— 鉴权 Spring Security
Executor._check_permission · agent.py:555

场景:这次 run 到底被授权能调哪些工具?
方法:执行前检查工具名是否在 allowed 白名单里,不在就抛 PermissionDenied。
为什么:给 agent 一个「沙箱」边界——哪怕大脑抽风想调危险工具,没授权也调不动。

Retry / backoff + 幂等 —— 把「执行」做扎实
代码:Executor.run agent.py:581-610 · 幂等 key 562
遇到什么
真实世界会偶发抽风(网络抖动、临时 503);而有副作用的操作(写文件)不能因为重试就写两遍。
用什么方法
重试:瞬时错误(TransientError)就退避等待后重试,最多 N 次。幂等:用 工具名+参数 算一个 hash 当 key,同样的调用第二次直接返回缓存、跳过副作用。
怎么处理
本项目故意给 write_note 注入「第一次必失败」:于是你能看到「第 1 次瞬时失败→退避→第 2 次成功」;之后再提交同一个写操作,幂等命中,真实写入次数保持 1 次不变。
为什么
没有重试,偶发故障就让整个任务失败;没有幂等,重试又会导致重复扣款/重复写入。两者必须配对。
🔗 对应 @Transactional / Spring Retry 模板:把重试、幂等这些「执行边界」收在一处。
Tracer + budget —— 运行日志和预算(注意:≠ Memory)
代码:agent.py:612-645

场景:你要 debug、要算成本、要防止跑太久烧太多。
方法:每步记一行(工具、是否成功、判决、伪 token);累计 cost 回写 state,供 Terminator 检查预算。
为什么和 Memory 分开:Trace 给人看(运维/调试),Memory 给 agent 看(决策依据)。你不会让 agent 把「自己花了多少 token」当成业务事实去回忆。

★ LLM 接缝 —— 整个项目唯一需要改成大模型的地方
Brain.decide
代码:agent.py:648-663
遇到什么
上下文已经拼好了(ctx),现在要产出一个决策:调哪个工具、传什么参。
用什么方法
现在:调用规则大脑 Policy.choose(ctx...),返回一个 Action。
将来接 LLM:把 ctx 序列化成 messages 发给 GPT/Claude,让它返回一段 JSON,再解析成 Action。
怎么处理
它是一层薄壳。入参 ctx 早已是「拼好的 prompt 材料」(system + 工作记忆 + 召回记忆 + 工具 schema),出参永远是一个 Action。
为什么这样设计
把「决策」隔离成单一接缝,意味着你接真实大模型时,控制环、生产边界、记忆——一行都不用改。这就是「关注点分离」在 agent 里的最大价值。
这回答了你最初的疑惑:「agent 到底怎么用大模型?」
大模型不负责执行、不负责记忆、不负责安全。它只在 ③ 这一步干一件事:看着拼好的上下文,吐出「下一步调哪个工具、传什么参」。 其余 10 步(拼上下文、护栏、审批、执行、重试、自评、记忆、trace)都是你写的确定性工程代码包在它外面。

本项目把这一步用规则写死,正是为了让你看清:去掉大模型,agent 的骨架依然完整。大模型只是那颗可替换的「大脑」。

3 · 跟着示例任务,把 11 步走一遍(带真实数据)

现在把前面所有组件串起来。任务:「查 btc_price 按 10 倍 写进笔记 portfolio」。 计划是 查事实 → 算公式 → 记笔记 三步,所以控制环会转 3 圈。

开跑前:
[INTENT] parse 得到 {key:btc_price, op:mul, factor:10.0, note_key:portfolio}
[PLAN] 查事实 → 算公式 → 记笔记

第 1 圈 步骤:查事实(lookup)

① 取步骤查事实(cursor=0)
② 拼上下文working=0(还没历史)、recalled=若干、schemas=3 个工具
③ 大脑决策Action("lookup", {key:"btc_price"}, risk=LOW)
④ 输入护栏通过(lookup 无危险参数)
⑤ 审批跳过(LOW 风险,不过门)
⑥ 执行鉴权✓ → 幂等未命中 → Router→tool_lookup → 返回 50000
⑦ 输出护栏通过
⑧ 自评continue(成功,且不是最后一步)
⑨ 工作记忆scratch["lookup"]=50000,cursor→1
⑩ 长期记忆记一条情节(成功才记)
⑪ trace记录 #1,累计 cost
[DECIDE] lookup[EXEC] 成功→50000[REFLECT] continue

第 2 圈 步骤:算公式(calc)

① 取步骤算公式(cursor=1)
③ 大脑决策注入上一步结果:a = scratch["lookup"] = 50000 → Action("calculator", {a:50000, op:"mul", b:10.0}, LOW)
④ 输入护栏检查操作数不为空、不超 1e9 → 通过
⑥ 执行tool_calculator(50000, mul, 10) → 500000.0
⑧ 自评continue
⑨ 工作记忆scratch["calc"]=500000.0,cursor→2
[DECIDE] calculator[EXEC] 成功→500000[REFLECT] continue

第 3 圈 步骤:记笔记(note)—— 高风险,点亮整条生产边界

① 取步骤记笔记(cursor=2,最后一步)
③ 大脑决策注入 scratch["calc"]=500000,拼出文本 → Action("write_note", {key:"portfolio", text:"portfolio = 500000 (来自 查 btc_price …)"}, risk=HIGH)
④ 输入护栏key=portfolio 合法 ✓;但文本 37 字 > 30 上限 → 截断改写([GUARD] 截断)
⑤ 审批HIGH → 过人审门,auto 放行([APPROVE])
⑥ 执行鉴权✓ → 幂等未命中 → 注入故障:第 1 次瞬时失败 → 退避 0.05s → 第 2 次成功 → 写入 memory.json → 幂等缓存登记
⑧ 自评done(成功,且已是最后一步)
⑨ 工作记忆scratch["note"]="已写入笔记[portfolio]",done=True
[GUARD] 37>30 截断[APPROVE] 通过[EXEC] 第1次失败→重试→成功[REFLECT] done

🏁 收尾 _finish(agent.py:739)

Terminatordone=True → 跳出 while,停止原因「goal 达成 / 收尾」
最终答案计算结果=500000,已写入笔记[portfolio]
[ANSWER] 计算结果=500000…RUN SUMMARY
你真正该带走的:每一圈,数据都是这样接力的——上一步的输出经由 scratch 工作记忆,注入下一步的参数。这就是 agent「多步链式调用」的底层机制。

4 · 三条分支场景(agent 的「应变能力」)

happy path 谁都会跑。真正体现 agent 价值的,是它出错时怎么自救。项目内置的 evaluate()(agent.py:772)就在测这几条。

① 改道(replan)

触发:查一个不存在的 key(如 doge_price)。
过程:lookup 抛 KeyError → error_handler 包成失败 → Reflector 判 replan → 下一圈 Planner.replan 把查询换成兜底 btc_price → 继续算和记。
结果:任务照样完成。这就是「出错改道」。

② 拦截(guard-block)

触发:要写的笔记 key 是 system(禁写黑名单)。
过程:④ 输入护栏抛 GuardrailBlock → 被主循环捕获、记为 BOUNDARY 失败 → Reflector 判 done(确定性拦截,重试无意义)。
结果:任务失败但安全,绝不会写进危险 key。

③ 预算终止(budget)

触发:把 max_steps 设成 1。
过程:跑完第 1 步后,Terminator 发现 steps_taken(1) ≥ max_steps(1) → 立即停。
结果:任务没做完就停。这是防失控的保险丝——宁可没做完,不能烧爆预算。

重试 + 幂等,单独再看一眼(最容易被忽视的细节)

示例任务里 write_note 被注入「第一次必失败」。配合幂等演示(demo_idempotency,agent.py:757):

第1次调用→ 瞬时失败→退避 0.05s→ 第2次调用→成功,写入1次
事后再提交同一个写操作→ 幂等 key 命中→跳过副作用→ 真实写入次数仍是 1

为什么这俩必须配对:光有重试,偶发故障会导致重复写入(想象重复扣款);光有幂等,遇到抖动又会直接失败。重试保证「最终成功」,幂等保证「只生效一次」。

5 · 五组最容易混的概念

学这套东西,80% 的困惑来自把下面几对当成一回事。各记两句话就够。

Guardrails vs ErrorHandler

ErrorHandler 管「崩了」:工具抛异常 → 包成 Result(ok=False),事后兜底。
Guardrails 管「合法但危险」:没崩,但越界/禁写/超长,需要事前拦或改。
一个是「正确性」,一个是「安全性」。

Trace vs Memory

Trace 给人看:耗时、token、cost,用来 debug 和算钱。
Memory 给 agent 看:它做决策的依据。
你不会让 agent 把「自己花了多少 token」当业务事实去回忆。

Retrieval vs Recency

Recency =「最近 N 条」(recent),按时间取,简单但可能全是近期噪声。
Retrieval =「最相关 K 条」(semantic_recall),按语义取。
项目特意写成两个方法,就为证明:取回 ≠ 看最近。

working / long-term / semantic memory

working=本次 run 的便签(State,内存,跑完即弃)。
long-term=跨进程的持久经验(memory.json)。
semantic=不是第三个仓库,是「查 long-term 的方式」。

Router 路由表 vs Tool schema

路由表只认名字:"write_note" → 那个函数,负责「怎么把选中的调起来」。
Tool schema 是工具自带的描述+风险,负责让大脑「选谁」。
schema 决定选谁,路由表决定怎么调。

规则大脑 vs LLM 大脑

本项目 Policy.choose 是写死的 if-else;真实 agent 这里是大模型。
但接口完全一样:吃 Context,吐 Action。
所以换大脑只动一个函数,这就是「接缝」的意义。

6 · 从 demo 走向真实系统:三个扩展点

① 加一个工具

1) 写个纯函数(出错就抛);
2) 在 _build_tools(agent.py:685)登记 Tool(名, 描述, 风险, fn);
3) 让某步会选中它。

它会自动享受路由、鉴权、重试、幂等、护栏、trace——因为这些都在执行链路上,不在工具里。

② 换成真实 LLM

唯一接缝在 Brain.decide(agent.py:648)。

把 ctx(已是拼好的 system+记忆+工具 schema)序列化成 messages 发给 LLM,让它返回 JSON,解析成 Action。

控制环、生产边界、记忆,一行都不用动。

③ 换成向量召回

只改 semantic_recall(agent.py:463):

把「词重叠打分」换成「query 过 embedding 模型 → 向量库里 ANN 检索 top-k」。

recent()(按时间)保持不变,正好对照出 retrieval 与 recency 是两条独立路径。

一句话收尾:读懂 agent.py 里 run() 的 ①~⑪ 十一步,你就读懂了「一个 agent 由哪些组件构成、谁调用谁」。
它不神秘:在你熟悉的 Controller→Service→DAO 外面,套一圈会自评、会改道、会判停的 while 循环(控制环),再把「决策」这一步留给大模型(LLM 接缝),周围用护栏、审批、重试、幂等、记忆把它兜住(生产边界)。 剩下的全是实现细节。
